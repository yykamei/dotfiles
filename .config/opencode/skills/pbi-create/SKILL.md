# プロダクトバックログアイテム作成

スクラムの**プロダクトバックログアイテム (PBI)** を、ユーザーとの対話を通じて作成するスキルです。

## 使い方

このスキルは 2 通りの起動方法に対応します。

### 1. 通常の対話で起動する

ユーザーが「PBI を作りたい」「バックログアイテムを起票したい」と
依頼してきた場合、エージェントがこのスキルをロードし、対話の冒頭で
ビジネス面での要求（実現したいこと・課題）をユーザーに尋ねます。

### 2. slash command として起動する（任意）

slash command を併設している環境では以下の形式が使えます:

```bash
/pbi-create [実現したいこと・課題]
```

例:

```bash
/pbi-create ワーカーが勤怠実績を自分で確認できるようにしたい
```

```bash
/pbi-create 企業が募集条件を柔軟に変更できるようにしたい
```

**ポイント**: 引数や依頼内容には「how（どう実現するか）」ではなく
「what/why（何を実現したいか・なぜ必要か）」を伝えてください。

## 出力フォーマット

最終的に以下のフォーマットで PBI を Markdown として出力します。

### 文体ルール

**出力する Markdown の本文（特に「概要」セクション）は、すべて「です・ます調」で記述すること。「だ・である調」は使わない。**

- 良い例: 「〜という課題があります。」「〜できるようにします。」「〜が必要です。」
- 悪い例: 「〜という課題がある。」「〜できるようにする。」「〜が必要だ。」

> 受入基準の項目は「〜できること」「〜が担保されていること」のような体言止め形式で記述するため、このルールの対象外です。起こったら困ることの箇条書きも「〜する可能性」「〜のリスク」のような名詞句や体言止めで記述してください。ただし、完全な文で書く場合は「です・ます調」に従ってください。

### テンプレート

```markdown
# 概要

[です・ます調で記述します。背景・課題・解決策を記述します。例: 「現状、〇〇という課題があります。これを解決するために、△△が□□できるようにします。」]

## ターゲットユーザー

[この変更で価値を受け取るユーザーを、です・ます調の自然な文章で記述します。]

## アウトカム

[この PBI が実現された場合の成功状態を、です・ます調の自然な文章で記述します。]

# 受入基準

- [ ] [体言止め形式。例: 「〜できること」「〜が担保されていること」]
- [ ] ...

# 起こったら困ること

- [簡潔な箇条書き]
- ...
```

### 出力例

以下は「です・ます調」で記述された PBI の具体例です。概要セクションの文体を参考にしてください:

```markdown
# 概要

現状、ワーカーは自分の過去の勤怠実績を確認する手段がなく、勤務時間や出勤日数について問い合わせが発生しています。これを解決するために、ワーカーが自分の勤怠実績一覧を画面上で確認できるようにします。

## ターゲットユーザー

この機能の主なターゲットユーザーは、自身の勤怠状況を把握したいワーカーです。

## アウトカム

ワーカー自身が勤怠状況をいつでも確認でき、問い合わせ対応の負荷が軽減されている状態を目指します。

# 受入基準

- [ ] ワーカーが過去の勤怠実績一覧を確認できること
- [ ] 勤怠実績が日付順に表示されること
- [ ] 対象期間を絞り込んで表示できること

# 起こったら困ること

- 他のワーカーの勤怠実績が閲覧できてしまう可能性
- 既存の勤怠登録機能へのデグレ
```

#### もう 1 つの例（企業向け）

```markdown
# 概要

現状、企業が一度公開した募集条件を変更するには、運営への問い合わせが必要であり、応募状況の変化に応じた柔軟な調整ができていません。これを解決するために、企業が自身で募集条件を編集できるようにします。

## ターゲットユーザー

このスキルが価値を提供する主なターゲットユーザーは、自社の募集条件を運用しながら最適化したい企業の採用担当者です。

## アウトカム

応募が想定より少ない・多いといった状況に対し、企業が自律的に募集条件を見直せる状態を目指します。あわせて、運営への変更依頼問い合わせが減少することも期待します。

# 受入基準

- [ ] 企業の採用担当者が公開済みの募集条件を編集できること
- [ ] 編集内容が応募者向け画面に正しく反映されること
- [ ] 編集履歴が運営側から追跡できること

# 起こったら困ること

- 既に応募済みのワーカーに不利な条件変更が行われる可能性
- 公開中の他社募集に意図せず影響が及ぶデグレ
```

### 各セクションのガイドライン

- **概要**: 背景・課題・解決策を**です・ます調の自然な文章**で記述します。加えて、以下のサブセクションを設けます:
  - **ターゲットユーザー**: この変更で価値を受け取るユーザーを、です・ます調の自然な文章で明記します
  - **アウトカム**: この PBI が実現された場合の成功状態を、です・ます調の自然な文章で明記します
- **受入基準**: 「満たしたい状態」を定義します。実装の詳細（API設計、DB設計、具体的なUI要素など）には踏み込みません。「〜できること」「〜が担保されていること」のような体言止め形式で記述します
- **起こったら困ること**: デグレや意図しない副作用を中心に、簡潔な箇条書きで記述します

## 実行フロー

```
Progress:
- [ ] Phase 1: 意図の深掘り
- [ ] Phase 2: 受入基準と起こったら困ることの整理
- [ ] Phase 3: PBI の出力とレビュー
```

### Phase 1: 意図の深掘り

ユーザーが与えた引数をもとに、PBI の「概要」セクションを書くために必要な情報を質問で明確にする。

以下の観点が不明確な場合、質問して確認する。関連する観点は `question` ツールを使って1回の質問にまとめてよい:

1. **What（何をしたいのか）**: 実現したい機能や変更の本質
2. **Why（背景）**: なぜ今それが必要なのか、どんな課題やペインがあるのか
3. **Who（対象ユーザー）**: この変更で価値を受け取るのは誰か（ワーカー、企業、社内オペレーターなど）
4. **Outcome（最終的なアウトカム）**: これが実現された場合、どんな状態になれば成功と言えるか

引数から十分に読み取れる情報については質問を省略してよい。すべてが明確な場合はこのフェーズをスキップできる。

### Phase 2: 受入基準と起こったら困ることの整理

Phase 1 で得た情報をもとに、受入基準と起こったら困ることを整理する。

**受入基準について**:

- まずエージェント側で受入基準の案を提示し、ユーザーに過不足を確認する
- 「満たしたい状態」として定義する（例: 「ワーカーが過去の勤怠実績一覧を確認できること」）
- 実装の詳細には踏み込まない（例: NG「REST API で GET /attendances を実装する」）
- 不足している観点があればユーザーに質問する

**起こったら困ることについて**:

- 既存機能へのデグレや意図しない副作用を中心にヒアリングする
- エージェント側でも想定されるリスクを提示し、ユーザーに確認する
- 箇条書きで簡潔に記述する

### Phase 3: PBI の出力とレビュー

- 出力フォーマットに従って PBI を Markdown で出力する
- **概要セクション（ターゲットユーザー・アウトカムのサブセクションを含む）は必ず「です・ます調」で記述する。出力前に文体を確認し、「だ・である調」の文が含まれていないことを確かめる**。具体的には、文末が「〜だ。」「〜である。」「〜する。」「〜なる。」「〜した。」「〜となる。」「〜必要だ。」などになっていないかチェックし、「〜です。」「〜します。」「〜なります。」「〜しました。」「〜必要です。」などに修正する
- ユーザーにレビューしてもらい、修正が必要な場合は対応する
- 出力先は `/tmp/` 配下とし、ファイル名は次のテンプレートに従う:

  ```
  /tmp/pbi-YYYYMMDD-<英数字スラッグ>.md
  ```

  - `YYYYMMDD` は作成日（ローカル日付）
  - スラッグはタイトルを英数小文字 + ハイフンで要約したもの（最大
    40 文字程度）。日本語タイトルの場合はローマ字または英訳した短い
    キーワードを用いる
  - 例: `/tmp/pbi-20260425-worker-attendance-view.md`,
    `/tmp/pbi-20260425-company-flexible-recruitment.md`
- 既存ファイルが存在する場合は上書きせず、末尾に `-2`, `-3` のような
  連番を付けて区別する

## インタラクションの原則（最重要）

このスキルでは、**ユーザーとの対話を最大限に重視**します。

- **推測せず必ず質問する**: AI エージェントはユーザーの背景情報を持っていない。曖昧な表現や複数の解釈が可能な場合は、自分の解釈で進めず必ず質問する
- **関連する質問はまとめる**: `question` ツールを活用し、関連する質問は1回の呼び出しにまとめて聞く。ただし、前の回答に依存する質問は分けて順番に聞く
- **納得するまで先に進まない**: すり合わせが不十分なまま次のフェーズに進まない。ユーザーの意図を十分に理解できたと確信してから PBI を生成する
- **Few-shot による改善に頼らない**: 出力後に修正を繰り返すのではなく、事前の質問で正しい方向性を確立する
